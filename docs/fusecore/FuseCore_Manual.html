◈Introduction

FuseCore is a B2B core banking API. It is never called directly by end-users or browsers — it is consumed exclusively by other backend services inside a secure network. Every request must carry a valid API key; optionally, calls are further restricted by IP allowlist.

Multi-Tenant

Each tenant gets an isolated MySQL database. All operations are scoped to the calling tenant — no data bleeds across tenants.

Double-Entry Ledger

Every financial movement posts balanced GL entries. The ledger is the source of truth for all money positions.

Loan Lifecycle

Full lifecycle: application → approval → disbursement → repayment → closure, with write-off, waiver, recovery, and foreclosure paths.

PII Encryption

BVN, NIN, email, phone encrypted at rest with AES-256-GCM. Searchable via HMAC-SHA256 blind indexes.

AML Engine

6-rule AML screening on every transaction. Triggered rules create an alert routed to the compliance review queue.

Async Workers

8 BullMQ queues handle notifications, audit logging, reporting, reconciliation, outbox dispatch, loan processing, and AML checks.

◈Local Setup

Prerequisites

  • Node.js ≥ 18, Yarn 1 (never npm or pnpm)
  • MySQL 8.x — one database for the management plane, isolated databases per tenant
  • Redis 7.x — shared by cache and BullMQ

First-Time Setup

# 1. Install dependencies
yarn install

# 2. Copy env template and fill in values
cp .env.example .env

# 3. Run migrations (management DB)
yarn db:migration:run

# 4. Start API and worker
yarn start:dev
yarn start:worker:dev
Never use DB_SYNCHRONIZE=true in production. All schema changes must go through TypeORM migrations. Synchronize mode is only acceptable against a local throw-away database.

◈Authentication

All domain routes require the X-API-Key header. The guard is global — registered as APP_GUARD in api.module.ts. Do not add @UseGuards(ApiKeyGuard) to individual controllers.

API Key Header

X-API-Key: <your-api-key>
X-Request-ID: <uuid-v4-request-id>  // optional; generated if missing or invalid

Managed API Keys

Tenant API keys are stored in tenant_api_keys as hashes. Each key has a name, prefix, status, scopes, optional expiry, optional per-key IP allowlist, and usage metadata. Plaintext keys are returned only once when created or rotated.

Key lifecycle events and denied API requests are persisted in the admin database security_audit_logs table.

Scopes

Keys may be granted * or explicit scopes such as customers:read, customers:write, accounts:read, accounts:write, transactions:read, transactions:write, payments:read, payments:write, charges:read, charges:write, reconciliation:read, reconciliation:write, webhooks:read, webhooks:write, ledger:read, ledger:write, loans:read, loans:write, products:read, products:write, reporting:read, metrics:read, aml:read, and aml:write.

IP Allowlist (optional)

Set per-key allowedIps to restrict a tenant API key. If a key has no per-key list, IP validation is skipped for that key. Admin routes use X-Admin-API-Key and do not perform IP allowlist checks.

Public Routes

Endpoints decorated with @Public() bypass API key validation. Currently only the health check (GET /health) is public.

No customer-facing login layer. FuseCore is a B2B service. Access is controlled with managed tenant API keys, scopes, expiry, revocation, request IDs, and IP allowlists.

Pagination, Filtering & Sorting

List endpoints use page, limit, sortBy, and sortOrder=ASC|DESC. Defaults are page=1, limit=20, and descending created-date order. limit is capped at 100. Unknown sort fields fall back to the endpoint default.

Standard sorting is enabled for customers, accounts, transactions, payments, loans, and loan repayments. Domain filters such as status, type, customerId, accountNumber, fromDate, and toDate remain endpoint-specific.

Money Movement Idempotency

All externally exposed money movement endpoints require the Idempotency-Key header. The key must be unique per business operation and no longer than 128 characters. Replaying the same key returns the original movement where applicable instead of posting money again.

Idempotency-Key: PAY-2026-000001

Protected routes: POST /transactions/transfer, POST /transactions/debit, POST /transactions/credit, POST /payments, PATCH /loans/:id/disburse, POST /loans/:id/repay, POST /loans/:id/recover, and PATCH /loans/:id/foreclose.

Transfer Concurrency Safety

Inside the transfer's database transaction, both source and destination accounts are locked with SELECT ... FOR UPDATE (pessimistic write lock) before any balance modification occurs. This prevents phantom balance reads under concurrent writes.

◈Loans

Full lending lifecycle from application through closure, with collateral, valuation history, restructuring, arrears aging, penalty interest, collections workflow, provisioning, write-off, recovery, and foreclosure paths.

Operations Coverage

  • Already present before this upgrade: application, approval, rejection, disbursement, schedules, repayments, write-off/undo, recovery, foreclosure, interest waiver, and GL posting for the core loan lifecycle.
  • Added: collateral entities, valuation records, restructuring/rescheduling history, arrears aging buckets, penalty interest, automated default detection, collections status workflow, provisioning fields, schedule-aware partial payment allocation, and penalty GL posting rules.

Loan State Machine

PENDING → UNDER_REVIEW → APPROVED → DISBURSED → ACTIVE → CLOSED
PENDING → REJECTED
ACTIVE → DEFAULTED
ACTIVE / DEFAULTED → WRITTEN_OFF → ACTIVE (undo write-off)
ACTIVE → FORECLOSED

Disbursement — Two-Phase Commit

Disbursement follows a two-phase pattern to preserve financial integrity:

  1. Phase 1 (DB transaction): Loan status updated to DISBURSED/ACTIVE, GL entries posted — all in a single atomic DB transaction.
  2. Phase 2 (credit): Account credit executed. If the credit fails, a compensating GL reversal and loan state rollback are immediately applied.

◈Ledger

A double-entry General Ledger (GL). Every financial movement in FuseCore produces an immutable posted journal batch with explicit debit and credit lines, so the books are always mathematically provable.

Double-Entry Accounting — the Concept

The foundational rule of accounting is the accounting equation:

Assets = Liabilities + Equity

Every transaction changes at least two accounts but leaves the equation balanced. The direction of the change depends on the normal balance of each account type:

Account TypeNormal BalanceIncreased ByDecreased By
AssetDebitDebitCredit
LiabilityCreditCreditDebit
EquityCreditCreditDebit
Revenue / IncomeCreditCreditDebit
ExpenseDebitDebitCredit

A Trial Balance is the periodic proof that this holds: the sum of all debit-side balances must exactly equal the sum of all credit-side balances. If they differ, a posting was made incorrectly.

Journal Posting Model

FuseCore stores accounting records in journal_batches and journal_lines. A batch/header carries the journal reference, posting date, value date, accounting period, source module, and source reference. Lines carry the GL account, side (DEBIT or CREDIT), amount, currency, and narration.

Posted journals must balance before they are saved. Once posted, monetary lines are immutable. Mistakes are corrected by posting a compensating reversal journal through /ledger/entries/reverse; the original journal remains intact for audit.

How FuseCore Posts Journals

Every financial operation (credit, debit, transfer, loan disbursement, repayment, write-off, recovery, foreclosure, charges/taxes, and fixed-deposit penalty) calls ledgerService.autoPost() internally. The call accepts the same EntityManager used by the balance-update transaction, making the journal write atomic with the account balance change — if either fails, both roll back.

◈AML / Compliance

Every transaction is evaluated asynchronously against the AML rules engine. Alerts are created for any triggered rule and routed to the compliance review queue.

Nigerian Regulatory Context

Nigeria's AML/CFT framework is governed by three interlocking instruments:

  • CBN AML/CFT Regulations 2022 — the Central Bank of Nigeria's primary rulebook for banks and fintechs. Mandates Customer Due Diligence (CDD), risk-based transaction monitoring, and reporting obligations.
  • Money Laundering (Prevention and Prohibition) Act 2022 (MLPPA) — the enabling legislation. Makes structuring, layering, and tipping-off criminal offences.
  • NFIU Guidelines — the Nigerian Financial Intelligence Unit (Nigeria's national FIU, a member of the Egmont Group) sets specific thresholds and report formats. All STRs and CTRs are filed via NFIU's goAML platform.

Nigeria is also a FATF member country, so the 40 FATF Recommendations apply — in particular Recommendation 1 (risk-based approach), which is why FuseCore applies tighter thresholds for VERY_HIGH risk-rated customers rather than a single fixed limit.

Key mandatory obligations for financial institutions:

  • Currency Transaction Reports (CTR) — any single cash transaction ≥ ₦5 M (individual) or ₦10 M (corporate) must be reported to NFIU within 24 hours.
  • Suspicious Transaction Reports (STR) — any transaction suspected to be proceeds of crime must be reported within 3 working days. The goAML reference number is stored in regulatorRef on the SAR.
  • Tipping-off prohibition — disclosing to a subject that they are under STR investigation is a criminal offence (CBN AML/CFT Regulations 2022, Reg. 15). The AML endpoints must never be exposed to customer-facing systems.
  • Record retention — transaction records and compliance reports must be kept for a minimum of 5 years.

AML Rules Engine

Six rules run in parallel on every transaction. If any rule triggers, a composite risk score is computed and an alert is created. The score starts at 25 (LOW), 50 (MEDIUM), or 75 (HIGH) based on the worst triggered severity, and rises by 5 for each additional triggered rule, capped at 100.

Rule CodeSeverityTrigger LogicRegulatory Basis
CTR_THRESHOLDHIGH Single transaction ≥ ₦5 M (individual) / ₦10 M (corporate) / ₦500 k (VERY_HIGH risk) NFIU CTR reporting obligation; tighter threshold for high-risk is the FATF risk-based approach
STRUCTURINGHIGH Rolling 24 h window sum is 90–100% of the applicable CTR threshold across ≥ 2 transactions MLPPA 2022 criminalises structuring ("smurfing") — intentionally splitting amounts to evade reporting
VELOCITYMEDIUM Debit-only: > 10 outflows in the last 60 min AND > 5× the account's 30-day hourly average Unusual outflow velocity is a key indicator of account takeover and mule account activity
ROUND_AMOUNTLOW Large transaction amount is an exact round number (e.g. ₦500,000.00 exactly) Round amounts are a common laziness pattern in illicit transfers; legitimate payroll/supplier payments rarely land on exact round figures
DORMANT_REACTIVATIONMEDIUM No transactions in the past 90 days, then a transaction of ≥ ₦1 M appears CBN defines dormancy at 6 months; FuseCore flags earlier at 90 days. Sudden large activity on a dormant account is a layering red flag
HIGH_RISK_COUNTERPARTYHIGH Counterparty account or customer is on the internal high-risk watchlist (PEPs, sanctioned entities) CBN CDD regulations require Enhanced Due Diligence for Politically Exposed Persons (PEPs) and sanctioned counterparties

Alert Lifecycle

An alert is created with status OPEN. A compliance officer then resolves it (false positive → CLOSED) or escalates it (ESCALATED, immutable). Escalated alerts are candidates for a Suspicious Activity Report (SAR).

SAR (Suspicious Activity Report) Lifecycle

A SAR groups one or more alerts against the same customer into a structured report for submission to NFIU via goAML. The lifecycle is DRAFT → FILED → ACKNOWLEDGED. Once filed, the regulatorRef field stores the goAML reference number. An acknowledged SAR is immutable.

Tipping-off prohibition. These endpoints are for internal compliance use only. Never expose them to customer-facing systems or return their data in any customer-visible API response. Disclosing the existence of an STR/SAR to the subject is a criminal offence under CBN AML/CFT Regulations 2022, Reg. 15.

◈Multi-Tenancy

FuseCore uses a database-per-tenant isolation model. Each tenant has its own MySQL database. The management database stores tenant metadata and credentials; it never stores customer or financial data.

How it Works

  1. Caller sends the tenant's X-API-Key header. ApiKeyGuard resolves the matching tenant and attaches it to the request.
  2. TenantDataSourcePool resolves the tenant's database connection from an in-memory pool or establishes a new one.
  3. All domain services are @Injectable({ scope: Scope.REQUEST }) — they receive the tenant-scoped DataSource via request context.
  4. Tenant databases are provisioned automatically on POST /tenants and are seeded with the default GL chart of accounts.
Cross-tenant data access is impossible by design. Services never share DataSource instances across tenants. The pool is keyed by shortname; each entry is an independent TypeORM DataSource pointing to a different database.

◈Durable Outbox

The durable outbox protects side effects that must happen after a successful database commit. Services write outbox_events rows inside the same transaction as the domain change. The worker publishes those rows after commit.

Why this exists. BullMQ retries jobs after they are enqueued, but it cannot protect the gap between a successful MySQL commit and a process crash before queue.add(). The outbox closes that gap.

Outbox Table

FieldPurpose
eventIdUnique idempotency key for the event and dispatched BullMQ job
eventTypeDomain event name such as transaction.completed or loan.disbursed
aggregateType / aggregateIdBusiness object that produced the event
payloadEvent body delivered to the dispatcher target
targetQueue / targetJobNameOptional BullMQ destination. Events without a target are still persisted as domain events.
statusPENDING, PROCESSING, PUBLISHED, FAILED, or DEAD_LETTER
attempts, nextAttemptAt, lastErrorRetry and diagnostic state

Dispatcher

The worker schedules DISPATCH_OUTBOX_EVENTS on the outbox queue every 10 seconds for each active tenant. The dispatcher scans due PENDING and FAILED rows, publishes to the configured BullMQ queue, and marks the row PUBLISHED. Failed dispatches use exponential backoff and move to DEAD_LETTER after maxAttempts.

Dispatched jobs use jobId = eventId, so queue insertion is idempotent for each outbox event.

Event Types

FuseCore defines event types for customer, account, transaction, payment, loan, ledger, and AML domains. The first production integration is on transfer/debit/credit: transaction domain events, audit log jobs, AML evaluation jobs, and transfer SMS jobs are written as outbox rows inside the same balance-update transaction.

// Transaction service pattern
await dataSource.transaction(async (manager) => {
  // write balances, transactions, charges, and ledger journals
  await outboxService.addEvent({
    eventType: "transaction.completed",
    aggregateType: "Transaction",
    aggregateId: transactionRef,
    targetQueue: "aml",
    targetJobName: "evaluate_aml_rules",
    payload: { transactionRef, accountId, amountKobo }
  }, manager);
});
Rollback invariant. Tests assert that if the transaction rolls back, no outbox row is written and no side effect can be dispatched.

◈PII Encryption

Encrypted Fields

The following customer fields are encrypted at rest using AES-256-GCM via the EncryptTransformer TypeORM column transformer:

email phone bvn nin identityNumber

Blind Indexes

Searchable encrypted fields have a parallel *_hash column containing an HMAC-SHA256 blind index. All WHERE clauses on these fields must target the hash column:

// ✓ Correct — query the blind index
const customer = await repo.findOne({
  where: { email_hash: blindIndex(email) },
});

// ✗ Wrong — querying encrypted ciphertext never matches
const customer = await repo.findOne({
  where: { email: email },  // always fails
});

Unique Constraints

Uniqueness is enforced on the *_hash columns, not the ciphertext columns. LIKE/fuzzy search on encrypted fields is not possible.

Key rotation requires a full re-encryption backfill of all rows with the new key. Plan for a maintenance window and data migration script. Rotation is a breaking change — the new ENCRYPTION_KEY must be deployed atomically with the backfill.

◈Financial Integrity Patterns

Pessimistic Locking

Balance-modifying operations use SELECT ... FOR UPDATE to prevent phantom reads under concurrent requests:

const lockedAccount = await manager
  .getRepository(AccountEntity)
  .createQueryBuilder('account')
  .where('account.id = :id', { id: account.id })
  .setLock('pessimistic_write')
  .getOne();

Two-Phase Financial Operations

Disbursement, repayment, recovery, and foreclosure all follow the same pattern:

// Phase 1 — durable state change (atomic DB transaction)
await dataSource.transaction(async (manager) => {
  await ledgerService.autoPost({ ... }, manager);
  await manager.getRepository(LoanEntity).update(id, { status: LoanStatus.ACTIVE });
});

// Phase 2 — external fund movement
const result = await transactionsService.credit({ ... });

// Compensating transaction on failure
if (!result.NO_ERROR) {
  await dataSource.transaction(async (manager) => {
    await ledgerService.autoPost({ /* reversed GL */ }, manager);
    await manager.getRepository(LoanEntity).update(id, { status: LoanStatus.APPROVED });
  });
}

Idempotency

Debit, credit, and transfer all accept an optional reference / transactionRef. Before executing, the service checks for an existing transaction with the same reference and returns the original result without re-executing.

Monetary Arithmetic

All balance calculations use decimal.js. Native JS floating-point is never used for money.

import Decimal from 'decimal.js';

const newBalance = new Decimal(account.availableBalance).minus(amount).toNumber();

All schema changes must go through TypeORM migrations. DB_SYNCHRONIZE must always be false in production.

# 4. Apply

# 5. Revert if needed

Migration files in libs/shared/src/database/migrations/ must be committed alongside the entity changes that require them. Never commit an entity change without its corresponding migration.

Migration API (production)

Migrations can be triggered at runtime via the Migration API. All four endpoints are protected by X-Admin-API-Key (same key used for tenant management).

Migrations run asynchronously. Once triggered, a Redis lock is acquired before the 202 response is returned — so traffic is blocked immediately. The lock is released automatically when the migration completes or fails. Each tenant DB tracks applied migrations independently in its own migrations table (TypeORM default), so only unapplied migrations run.

Lock behaviour during migration: While an admin DB migration is running, all /admin/tenants/* requests return 503 Service Unavailable. While a tenant DB migration is running, all API requests authenticated with that tenant's key return 503 Service Unavailable, and tenant-scoped worker jobs wait until the migration lock is released before touching that tenant database. The migration endpoints themselves (/admin/migrations/*) are always reachable so status can be polled and a conflicting trigger returns 409.

Soft Delete Convention

Every mutable entity carries @DeleteDateColumn({ name: 'deleted_at' }) deletedAt: Date. TypeORM automatically filters soft-deleted rows in all find* queries. To intentionally query deleted rows, pass { withDeleted: true }.

Never call repository.delete(id) or repository.remove(entity). Always use repository.softDelete(id). The single exception is AuditLogEntity, which has no deletedAt because audit records are immutable and must never be removed.
